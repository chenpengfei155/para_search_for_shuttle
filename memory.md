## 1. 目标定义 (Goal A vs Goal B)

### 1.1 安全目标 (Target)

对每个目标安全级别 `target ∈ {128, 256, 512}`，**固定** 对应的多项式环维度：

| target | n     | LWE 维度 n_lwe = ell·n | SIS 维度 n_sis = m·n |
|--------|-------|------------------------|-----------------------|
| 128    | 256   | 256·ell                | 256·m                 |
| 256    | 512   | 512·ell                | 512·m                 |
| 512    | 1024  | 1024·ell               | 1024·m                |

**搜索的自由参数**: `q (NTT 素数), ell, m, sigma, alpha_h`。

### 1.2 三个安全估计 (LWE / SIS_UF / SIS_sUF)

签名方案的安全性由 **三个并列的格难题** 共同保证。代码 (`para_alg_impl.py`) 调用 `lattice-estimator` 计算每个的"rough"比特安全性：

| 比特安全 | 攻击对象 | 维度 | 长度界 β | 含义 |
|---|---|---|---|---|
| **`LWE_security_bit`**     | 决策 LWE        | `n_lwe = ell·n` 维, `m_lwe = m·n` 样本, 模 q, 噪声 σ | — | **签名 secret key 的保密性** |
| **`SIS_UF_security_bit`**  | SIS (length=bv)   | `n_sis = m·n` 维, `m_sis = (ell+1+m)·n` 样本, 模 q | **β = bv**       | **存在性不可伪造 (EUF-CMA)** |
| **`SIS_sUF_security_bit`** | SIS (length=2·bv) | 同上                                                | **β = 2·bv**     | **强不可伪造 (sEUF-CMA)** |

其中 `bv = bs(r) + sqrt(n·m) · (alpha_h/4 + 1)`，由 σ、ell、m、α_h 共同决定。

**关键事实**：`SIS_sUF` 用了 **2·bv** 作为长度界——长度界更松意味着攻击更容易，所以恒有
```
SIS_sUF_security_bit  ≤  SIS_UF_security_bit
```
（同一组参数下 sUF 永远 ≤ UF。命中 sUF 比命中 UF 更难。）

### 1.3 两个搜索目标 (Goal A / Goal B)

我们要求 **LWE 与 SIS 同时落在目标带** `[target+5, target+12]`（用 +5 与 +12 留出对估计器误差的容差，同时不过度浪费比特）：

- **Goal A (UF goal)**:
  ```
  target+5 ≤ LWE_security_bit  ≤ target+12
  target+5 ≤ SIS_UF_security_bit ≤ target+12
  ```
  → 签名达到 **存在性不可伪造**，且 LWE/SIS 两侧安全均衡（不浪费 bits）

- **Goal B (sUF goal)**:
  ```
  target+5 ≤ LWE_security_bit   ≤ target+12
  target+5 ≤ SIS_sUF_security_bit ≤ target+12
  ```
  → 签名达到 **强不可伪造**（更强的安全性，但参数更难找）

**注意**：Goal B 不要求 SIS_UF 也在带内（实际上 Goal B 命中时 SIS_UF 通常远大于带 —— 这没问题）。同理 Goal A 不要求 SIS_sUF 在带。

**目标带** 详细列表：

| target | LWE / SIS 带 [target+5, target+12] |
|--------|------------------------------------|
| 128    | [133, 140]                         |
| 256    | [261, 268]                         |
| 512    | [517, 524]                         |

### 1.4 评估指标 (Bytes)

每条记录都计算三个 byte 度量：
- `PkBytes` (公钥字节) = `n/2/8 + ⌈n·m·⌈log₂ q⌉/8⌉`
- `SignBytes` (签名字节) = 由 `σ, r, α_1, ell·n, hh` 经熵公式算出
- `CombinedBytes` = Pk + Sign （**最小化目标**）

## 4. 算法内部 (Cheat Sheet)

来自 `para_alg_impl.py` 的 `compute_parameters`：

```
bk      = σ · √((ell+m)·n)
α_1     = ceil_pow2(√n · σ)              -- 2 的幂
scale   = R_SCALING[n/2]                 -- 2.75 / 2.78 / 2.81 for n=256/512/1024
r       = ⌈scale · √(α_1²/ell + bk²)⌉
mu_s    = n·(r/α_1)² + (ell+m)·n·r²
v_s     = 2n·(r/α_1)⁴ + 2(ell+m)·n·r⁴
bs      = √(mu_s + 6.13·√v_s)
bv      = bs + √(n·m) · (α_h/4 + 1)
σ_h     = 2r/α_h                         -- 必须 ≥ 0.05 (硬约束)
```

LWE / SIS 实例化:
```
LWE :    dim = ell·n,         samples = m·n,     mod q,  Gaussian σ
SIS_UF : dim = m·n,           samples = (ell+1+m)·n, mod q, β = bv
SIS_sUF: dim = m·n,           samples = (ell+1+m)·n, mod q, β = 2·bv
```

Bytes:
```
PkBytes       = ⌈n/2/8⌉ + ⌈n·m·⌈log₂ q⌉/8⌉
SignBytes     = (n/2 + n/2·log₂(2πe·(r/α_1)²) + ell·n/2·log₂(2πe·r²) + n·m·hh) / 8
CombinedBytes = PkBytes + SignBytes
```

## 5. 各杠杆 (Lever) 影响

| 杠杆 | LWE | SIS (UF/sUF) | 备注 |
|---|---|---|---|
| **σ↑** | ↑ (噪声更大) | ↓ (r,bs,bv 增大) | 经典 LWE-SIS tension |
| **q↑** | ↓ (易) | ↑ (β/q 比小，SIS 反而难) | 实测 q↑ → LWE 与 SIS 同方向反向 |
| **ell↑** | ↑↑ (n_lwe 直接增) | ↓ (bk↑→r↑→bv↑) | 整数变化大，主旋钮 |
| **m↑** | ↓ (样本多) | 复杂：n_sis↑(↑ SIS) 但 bv↑(↓ SIS) | m=1 让 n_sis 减半，是 t=512 Goal A 关键 |
| **α_h↑** | (无影响) | ↓ (bv↑) | "knee"型，到 σ_h≥0.05 上限处 SIS 急跌 |

**约束**：α_h 必须是 2 的幂，且 σ_h = 2r/α_h ≥ 0.05 → `α_h ≤ 40·r`

## 6. 通用调参策略 / 经验法则

1. **n 由 target 决定** (固定): 128→256, 256→512, 512→1024。
2. **ell 是 LWE 安全的主旋钮**: n_lwe = ell·n 决定大局；从 ell=3 起步，t=512 Goal A 可降 ell=2 或 m=1。
3. **α_h knee 效应是命中关键**:
   - σ 不动时 α_h 加倍 → SIS_UF 降 ~1 bit (温和)
   - 但 α_h 接近 σ_h 约束上限 (40·r) 时 SIS 急降 (knee)
   - **t=128 sUF / t=256 Goal A / t=512 Goal A 都靠 knee 实现**
4. **大 σ 解锁更大 α_h_max**: σ_h = 2r/α_h, σ↑ → r↑ → α_h_max↑（但 σ 也升高 Sign）
5. **bytes 优化方向**:
   - PkBytes 只看 (n, m, ⌈log₂ q⌉) → 小 q 直接降字节
   - SignBytes 主要看 r²（即 σ 与 ell+m 的乘积）→ 小 σ + 小 ell+m
   - 但小 σ + 小 q 会让 LWE 跌出 band → 需要在 Pareto 边界精细 sigma 调整 (0.005-0.01 步长)

### 已踩过的坑 (DO NOT REPEAT)

1. **m=1 + n=256/512**: SIS dim 太小 (n_sis=256/512)，SIS_UF 上限 ~80/~150 bit。**t=128/256 Goal B 不能用 m=1**
2. **t=512 用 ell=1**: LWE 直接崩到 ~120 bit（远低于带）
3. **q < 2^13 + 高 target**: LWE 反而过高（小 q 让 q/σ 比变小 → LWE 难度上升 → 远过带）
4. **早期 `EARLY_STOP_MARGIN=20` 过激**: 任一组合所有安全都 > target+20 就杀全组 → 误杀 alpha_h knee 区。现已设为 10000 (基本禁用)，靠 sub-prune 控制
5. **试图用 alpha_h 调到中间值**: α_h 必须 2 的幂，{1024, 2048, 4096, ...} 之间没空间。常见情况：α_h=4096 时 UF 在带，α_h=2048 时 UF 远过带，无中间值
6. **过度细 sigma 网格**: estimator 输出 LWE 是离散的（block-size 跳跃），sigma 步长 < 0.005 通常无新信息
7. **`t=128 / Goal A` 的 `q=7681, α_h=128` 848-byte pk 桶**: `Comb=1859` 与 `Comb=1864` 之间只有 UF 阈值切换，没有重叠区。`σ≈0.5072` 前后是一个干净断点，继续在这条线细抠只会重复看到 `UF=140.452` 与 `UF=139.868` 两个平台。
8. **`t=256 / Goal A` 的低 q 15-bit pk 桶 (`q≈31k..32k`)**: `α_h=1024` 时 UF 系统性过高，`α_h=2048` 时又系统性过低，两个 2 的幂之间没有可用中间值；邻近 `(ell,m)` 也没有救场。
9. **`t=256 / Goal A` 的 `q=57089, α_h=2048` 16-bit pk 桶**: `4055` sign 桶与 `4059` 命中之间也没有重叠区。LWE 在 `σ=0.6352..0.63595` 上几乎完全平到 `260.756`，sign jump 先于 LWE jump 发生。
10. **从现在起，`sigma` 的搜索步长最小固定为 `0.05`**: 不再做更细的 `0.01/0.005/0.001` 级微调。后续只允许结构性跳变：换 `q` 桶、换 `α_h`、换 `(ell,m)`，或者在粗 sigma 网格上移动。
11. **手工列 `q` 窗时，`q ≡ 1 mod (n/2)` 还不够，必须额外检查 primality**: `131329, 131585, ...` 这类 `256k+1` 数里大部分仍是复合数。后续搜索必须只保留真正的 NTT prime；`param_search.py` 现已在启动时强制校验这一点。
12. **`t=256 / Goal A` 的 `(ell,m)=(3,1)` 路线**: 22-27 bit 的 prime q 桶在粗 sigma 网格下会反复落到同一个 `~250 bit` 平台，典型值是 `LWE≈UF≈249.95`。继续在这条线上换同桶 q 或邻桶 q，只会在 `Comb≈3767..4012` 的近邻里重复看到 `11-16 bit` 的系统性缺口。
13. **`t=256 / Goal A` 的 `(ell,m)=(2,2)` 路线**: 即便把 `σ` 抬到 `2.0-2.5`、并把 q 拉到 23-bit，最好点也仍停在 `L/U≈247/248`。这条线没有隐藏的高-sigma rebalance，后续无需重扫。

## 7. 工程注记

### 7.1 环境

- Python: `python` (内含 sage)
- estimator 路径: `.deps/lattice-estimator/`
- 单次 `compute_parameters` 耗时: 0.1s (overshoot 区) → 80s (在 band 附近"真攻击"区)

### 7.2 CPU 利用率诊断 (15% 现象)

实测 60 任务 (24 慢) 不同 worker 数:
- 13 workers: 128s (默认)
- 20 workers: 121s
- 28 workers: 112s

`workers=20` 是甜区。瓶颈：慢任务集中在 band 附近，少数任务卡住其他空闲 worker。

设置：
```python
os.environ.setdefault("OMP_NUM_THREADS", "1")  # 等 BLAS 线程降为 1
# ProcessPoolExecutor(max_workers=20)
```
